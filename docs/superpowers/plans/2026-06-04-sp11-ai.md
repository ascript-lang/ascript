# SP11 — `std/ai` — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development
> (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use
> checkbox (`- [ ]`) syntax for tracking.

**Goal:** Ship `std/ai` — a unified LLM client wrapping the `genai` crate, covering all v1 providers
(OpenAI + OpenAI-compatible swarm + Anthropic + Gemini + **Bedrock SigV4 + Vertex ADC + Azure
OpenAI**), with `"provider:model"` selection + env creds, Tier-1 non-streaming, generator streaming,
class/`std/schema` structured output + a JSON-Schema projector, an in-interpreter tool loop, and OTel
GenAI spans through the SP12 soft hook (opt-in). Off-by-default `ai` feature; builds under
`--no-default-features`.

**Architecture:** Six phases. **A** is a **blocking `!Send` de-risk spike** (verify genai runs on the
LocalSet, else stand up the worker bridge) + the feature scaffold. **B** crate integration +
provider/credential plumbing + text generation (non-streaming) across the provider tiers. **C**
streaming (generators + `for await`). **D** structured output + the JSON-Schema projector. **E**
tools (the in-interpreter loop). **F** telemetry GenAI spans + embeddings + docs + holistic review.
Each phase is TDD over **fixture-replay** tests (no network/secrets), ends green on both feature
configs + clippy, and gets an independent review.

**Tech Stack:** Rust. `genai` (optional dep). Stdlib module over `Value`; stateful handles in
`Interp.resources`. `!Send` current-thread runtime + `LocalSet`; take-out-across-await; never hold a
`RefCell`/`resources` borrow across `.await`. Structured output reuses `validate_into` + `std/schema`.

**Spec:** `docs/superpowers/specs/2026-06-04-sp11-ai-design.md`.
**Depends on:** SP12 (`std/telemetry`) merged first — this plan consumes `Interp::telemetry_*`.

**Branch:** `feat/sp1-engine-parity` (per task context). Ships after SP12.

## Conventions for every task

- **Fixture-replay harness (no network, no secrets):** inject a custom reqwest client / send-seam
  into the genai `Client` that serves recorded fixtures from `tests/fixtures/ai/<provider>/<case>.{json,sse}`.
  Unit tests run `.as` programs through `ascript::run_source` against the replay client and assert the
  resulting `Value`. **No default test opens a socket or reads a key.** Model the streaming side on
  `net_http.rs`'s `SseState`/`SseReader`.
- **Read-before-write:** read `src/stdlib/net_http.rs` (reqwest client, SSE reader, native handles,
  take-out-across-await), `src/stdlib/schema.rs` + the `validate_into` core in `src/interp.rs`
  (structured output), `src/stdlib/log.rs` (module shape), SP12's `Interp::telemetry_*` hook, and the
  `genai` docs/examples for `ChatRequest`/`ChatOptions`/`exec_chat`/`exec_chat_stream`/`embed`.
- **Gate after each phase (paste tails):**
  `cargo test --features ai --test ai 2>&1 | tail`;
  `cargo test 2>&1 | tail` (0 failures, ai OFF by default);
  `cargo test --features ai 2>&1 | tail` (0 failures);
  `cargo build --no-default-features 2>&1 | tail` (builds, module cfg'd out);
  `cargo clippy --all-targets 2>&1 | tail` AND `cargo clippy --no-default-features --all-targets 2>&1 | tail` (clean);
  `cargo clippy --features ai --all-targets 2>&1 | tail` (clean);
  `grep await_holding_refcell_ref Cargo.toml` (still `deny`).
- **No VM/tree-walker differential cases** — `std/ai` is engine-agnostic stdlib (spec §5). Do not add
  `vm_differential` cases.

## Phase A — `!Send` de-risk spike + feature scaffold (BLOCKING)

**Files:** `Cargo.toml` (feature + dep); `src/stdlib/ai/mod.rs` (skeleton); `src/stdlib/mod.rs`
(register); a throwaway spike test. **This phase gates the whole sub-project — do it first.**

### Task A1: prove genai runs on the current-thread LocalSet

- [ ] **Step 1 — Add dep + feature** (behind `ai`, off `default`): `genai = { version = "0.6",
  optional = true }`; `ai = ["data", "net", "dep:genai"]`. `cargo build --features ai` compiles.
- [ ] **Step 2 — Spike test** (`tests/ai.rs`, `--features ai`, env-gated `ASCRIPT_AI_SPIKE=1` so it
  doesn't need a key — or use the fixture client): construct `genai::Client`, call `exec_chat`
  inside `tokio::task::spawn_local(...)` on a `current_thread` runtime + `LocalSet` with an
  `Rc<()>` in scope (proving `!Send` coexistence). Drive it to completion. **Assert it compiles and
  runs.** If it fails to compile with a `Send` bound error → go to Step 3 (bridge); else skip Step 3.
- [ ] **Step 3 — (ONLY if Step 2 fails) worker bridge** (`src/stdlib/ai/bridge.rs`): a dedicated
  `std::thread` running its own current-thread tokio runtime hosting the genai client; communicate via
  channels of `Send` plain-data request/response structs (never `Value`/`Rc` across the boundary);
  stream over an `mpsc`. Re-run the spike against the bridge. Document which path was taken in a
  module comment.
- [ ] **Step 4 — Decision record:** add a short note to the spec's §1 (or a sibling ADR
  `docs/superpowers/specs/adr/2026-06-04-genai-send-bridging.md`) recording the empirical result
  (in-LocalSet vs worker) so downstream phases know which path is live.
- [ ] **Step 5 — Commit:** `feat(ai): genai dependency + !Send spike (records the bridging decision)`.

### Task A2: module skeleton registered in all configs

- [ ] **Step 6 — Create `src/stdlib/ai/mod.rs`** with `exports()` (bindings `provider`, `generate`,
  `stream`, `embed`, `embedMany`, `tool`, each `super::bi("ai.<name>")`) and a `call(...)` stub
  (Tier-2 "not yet implemented" for unimplemented funcs). Register `#[cfg(feature = "ai")] pub mod
  ai;` + both router arms in `src/stdlib/mod.rs`.
- [ ] **Step 7 — Run:** `cargo build --features ai`, `cargo build`, `cargo build
  --no-default-features` all compile. Phase-A gate. Commit: `feat(ai): scaffold std/ai module + routing`.

## Phase D — structured output + JSON-Schema projector

**Files:** `src/stdlib/ai/json_schema.rs` (new), `request.rs`/`response.rs`, `mod.rs`. Reuses
`validate_into` (`src/interp.rs`) + `std/schema` (`src/stdlib/schema.rs`).

### Task D1: the class/schema → JSON-Schema projector (standalone, no network)

- [ ] **Step 1 — Failing unit tests** (Rust, no network): project a class `FieldSchema` →
  JSON Schema (required `id: number`, optional `name: T?`, nested class, `array<Class>`,
  `map<K,Class>`, defaults); project a `std/schema` tagged object → JSON Schema
  (`object`/`array`/`string`+minLength/pattern/`number`+min/max/`oneOf`/`union`/`optional`). Assert
  the emitted JSON Schema shape.
- [ ] **Step 2 — Implement** `json_schema.rs`: `class_to_json_schema(&Class)` and
  `schema_value_to_json_schema(&Value)`. (Read the `FieldSchema`/`validate_into` definitions and
  `std/schema`'s tagged-object kinds first.)
- [ ] **Step 3 — Run** → green. Commit: `feat(ai): class/std-schema → JSON-Schema projector`.

### Task D2: `shape:` wiring (class + schema), failure fusion

- [ ] **Step 4 — Failing tests:** `ai.generate({model, prompt, shape: Recipe})` (fixture returns
  JSON) → `[Recipe instance, nil]`, fields type-checked; `shape: schema.object({...})` → validated
  object; a fixture returning JSON that violates the shape → ONE Tier-1 `[nil, err]` (decode/shape
  fusion, like `resp.json(Class)`); a fixture returning non-JSON for a `shape:` call → Tier-1 err.
- [ ] **Step 5 — Implement** the `shape:` path: project → set genai's structured-output config per
  provider (OpenAI `response_format: json_schema`, Gemini `responseSchema`, Anthropic tool-forcing/
  JSON); **fallback** for any provider genai doesn't cover in structured mode = embed the JSON Schema
  in the prompt + `validate_into` on the returned text (spec §7.2 — documented, not silent). Run the
  result through `validate_into` for both class and schema shapes, fusing failures into one Tier-1
  pair.
- [ ] **Step 6 — Run** → green. Phase-D gate. Commit: `feat(ai): structured output (class + std/schema) with fused Tier-1 errors`.

## Phase E — tools (in-interpreter loop)

**Files:** `src/stdlib/ai/tools.rs`, `request.rs`/`response.rs`, `mod.rs`.

### Task E1: `ai.tool` + the sequential tool-use loop

- [ ] **Step 1 — Failing tests** (fixtures: a turn-1 response with a tool call, a turn-2 final
  answer): define `ai.tool({description, input: schema, execute})`; `ai.generate({..., tools:
  {weather}, maxSteps: 5})` → the loop validates args against the tool schema, calls `execute`, feeds
  the result back, and returns `out.text` (final) + `out.steps` (per-turn); a tool `execute`
  returning `[nil, err]` is fed back to the model as a tool-result (recoverable, not an abort);
  `maxSteps` caps the loop; a malformed tool definition (no `input` / `execute` not callable) →
  Tier-2 panic. An async `execute` is awaited.
- [ ] **Step 2 — Implement** `tools.rs`: tool-definition validation; the **sequential** loop inside
  `std/ai` (genai returns tool calls → validate args via the projector/`validate_into` → call
  `execute` (await if async) → append tool-result message → re-call genai → repeat until a final
  answer or `maxSteps`). Tool errors become tool-result content, not panics.
- [ ] **Step 3 — Run** → green. Phase-E gate. Commit: `feat(ai): tool calling + sequential tool-use loop (maxSteps)`.

## Self-review (author)

**Spec coverage:** §1 crate + `!Send` → Phase A (front-loaded blocking spike + bridge fallback); §2.1
providers/creds → B1; §2.2 text → B2; §2.3 streaming → C; §2.4 structured + projector → D; §2.5 tools
→ E; §2.6 messages → exercised in B2/E fixtures; §2.7 embeddings → F1; §3 telemetry spans → F2; §5
testing (fixtures + env-gated live) → every phase; §6 file map → followed. All covered.

**`!Send` discipline:** Phase A empirically resolves genai's Send-ness and records the path; every
genai await uses take-out-across-await; no `RefCell`/`resources` borrow across `.await`; the worker
bridge (if needed) carries only `Send` plain data, never `Value`/`Rc`. The one genuine risk (genai
futures requiring `Send`) is the first thing built and has a fully-specified fallback.

**Independence from telemetry:** F2 consumes only `Interp::telemetry_*` (SP12-owned, always-present
inert-when-off signatures); `cargo build --features ai` with telemetry OFF must compile — explicitly
gated in F2 step 4. No `ai ⇒ telemetry` Cargo edge.

**No placeholders:** every task has concrete `.as`/Rust assertions, exact paths, fixture locations,
and commands. Deferred-to-implementer details: exact genai API spellings (`ChatRequest`/`ChatOptions`/
`exec_chat_stream` signatures) — the implementer reads the genai docs (cited in the design) and the
recorded provider fixtures; the projector's exact JSON-Schema emission is pinned by the standalone
D1 unit tests. `cargo build` (not `test`) used for `--no-default-features` since the module is cfg'd
out there.
