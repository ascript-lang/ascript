# SP11 — `std/ai` (multi-provider LLM client) — Design

> **Status:** approved design, ready for implementation planning (superpowers:writing-plans).
> **Sub-project of** the post-cutover stdlib-expansion program. **Depends on** SP12 (`std/telemetry`)
> for the soft tracing hook it emits through — SP12 ships first. See
> `docs/superpowers/specs/2026-06-04-sp12-telemetry-design.md`.
> **Derived from** `docs/superpowers/specs/2026-06-04-sp11-sp12-ai-telemetry-PROPOSAL.md`; this
> design **overrides the proposal's "build native adapters" recommendation per the owner's decision
> to wrap a Rust crate** (`genai`), and adds the full provider set (incl. Bedrock + Vertex) in v1.

**Goal:** Add `std/ai` — a unified, AScript-idiomatic LLM client over **all** the v1 providers:
OpenAI + every OpenAI-compatible endpoint (Ollama, LM Studio, OpenRouter, LiteLLM, xAI/Grok,
DeepSeek, groq, together, Azure OpenAI key-auth) + Anthropic + Google Gemini + **AWS Bedrock (SigV4)**
+ **GCP Vertex AI (ADC/OAuth)** + **Azure OpenAI / AI Foundry**. Surface: `"provider:model"` selection
with env-default credentials; non-streaming → Tier-1 `[value, err]`; streaming → generators +
`for await`; structured output → class/`std/schema` + `validate_into` + a class/schema→JSON-Schema
projector; tools = AScript fns + schema, looped with `maxSteps`. `std/ai` emits OTel GenAI-convention
spans **through the SP12 soft hook** (opt-in, active only when telemetry is initialized).

**Architecture:** A new stdlib module `src/stdlib/ai/` (native Rust over `Value`) that **wraps the
`genai` crate** (jeremychone/rust-genai), `#[cfg(feature = "ai")]`-gated, registered in both arms of
`src/stdlib/mod.rs`, declared in `Cargo.toml [features]`. The crate's client + futures run on the
**current-thread `LocalSet`** with the take-out-across-await native-resource discipline; if `genai`'s
public futures turn out to require `Send`, a documented **dedicated single-thread worker** bridges the
boundary (§5). Streaming results are exposed as the established native-handle/generator shape (the SSE
idiom from `net_http.rs`). Stateful handles (provider/model/stream) live in `Interp.resources`.

**Tech stack:** Rust. `genai` (multi-provider, native protocols + OpenAI-compat, **bedrock_sigv4** +
**vertex** adapters). Async tree-walker + VM, both `!Send` (current-thread tokio + `Rc`/`RefCell`,
`LocalSet`). JSON via `stdlib::json`. Structured output via existing `validate_into` + `std/schema`.

---

## Non-goals (explicitly out of SP11 v1)

- **No streaming structured output** (`partialOutputStream`). Structured output is **non-streaming
  only** in v1 (proposal §2.4). Streaming yields text/tool-call/finish chunks, not partial typed
  objects.
- **No parallel tool execution.** The tool-use loop runs tool calls **sequentially** in v1 (a
  provider may emit several at once; we execute them in order). Parallel execution is a follow-up.
- **No native re-implementation of provider protocols.** We **wrap `genai`** (owner decision #1); we
  do not hand-write per-provider request/response adapters. (Contrast SP12, which hand-rolls 3
  trivial HTTP exporters — see SP12 §6 for why the wrap-vs-build call differs by module.)
- **No fine-tuning / files / assistants / batch APIs.** Chat, stream, structured output, tools,
  embeddings only.
- **Not in `default` features.** Network + new heavy dep → opt-in (`ai` feature, off by default).
  Must build clean under `--no-default-features` (feature off).
- **No hard Cargo dependency on `telemetry`.** GenAI spans go through the SP12 runtime soft hook;
  `std/ai` builds with telemetry absent.

## §1 — Crate choice & the `!Send` analysis (the critical decision)

**Decision: wrap `genai` (jeremychone/rust-genai), per the owner's wrap-a-crate preference.** It is
the single crate with the broadest coverage + active maintenance that makes the full v1 provider set
tractable:

- **Provider coverage (the deciding factor).** `genai`'s out-of-the-box adapters include `openai`,
  `anthropic`, `gemini`, `ollama`, `groq`, `together`, `fireworks`, `deepseek`, `xai`, `cohere`,
  `open_router`, `nebius`, `moonshot`, … **plus `vertex` and `bedrock_sigv4`/`bedrock_api`** — i.e.
  the OpenAI-compatible swarm **and** the two hard-auth providers (Bedrock SigV4, Vertex ADC/OAuth)
  the owner requires in v1, in one crate. It uses each provider's **native protocol** when available
  and the OpenAI-compat layer otherwise — exactly the architecture the proposal validated, but as a
  maintained dependency rather than code we own. Active maintenance: v0.6.x released 2026-05.
- **Why not the alternatives.** `async-openai` is OpenAI-only (no Anthropic/Gemini/Bedrock/Vertex).
  `rig-core` is an agent framework — broad provider list but it drags an agent/RAG abstraction layer
  and a heavier surface we do not want for a thin stdlib client. `genai` is the minimal multi-provider
  client. (Native hand-written adapters — the proposal's original pick — are rejected per owner
  decision #1 because re-implementing ~25 protocols + SigV4 + Vertex OAuth is the larger, riskier
  surface.)
- **Custom reqwest injection.** `genai` lets you inject a custom `reqwest::Client`
  (`ClientBuilder`/`WebConfig`), which we use to (a) **reuse our pooled rustls client** from
  `net_http.rs` where possible and (b) install the **injectable HTTP-send seam** for fixture-replay
  testing (§6). It exposes chat, streaming chat, tools, and embeddings.

### The `!Send` constraint — analysis, verification step, and bridging fallback

AScript's runtime is `!Send`: `#[tokio::main(flavor = "current_thread")]` + a `LocalSet`, with `Rc`/
`RefCell` interior mutability and the invariant **never hold a `RefCell` borrow across `.await`**
(`await_holding_refcell_ref = "deny"`). The risk is that `genai`'s futures assume a `Send`,
potentially multi-thread runtime.

- **Most tokio/reqwest crates run fine on a current-thread runtime** — `genai` is reqwest-based and
  its examples are plain `#[tokio::main]`; nothing in its API obviously requires a multi-thread
  runtime. **But the `Send`-ness of `Client::exec_chat` / `exec_chat_stream` futures is NOT
  documented and MUST be verified empirically** before building on it. This is the one load-bearing
  unknown in SP11.
- **Verification gate (Phase A, blocking).** A spike compiles a minimal call:
  `genai::Client::default().exec_chat(model, req)` driven by `tokio::task::spawn_local(...)` inside a
  `LocalSet` on a current-thread runtime, with our `Rc<Interp>` in scope. **Pass** = it compiles and
  runs without a `Send` bound error and we can hold the genai call's future across our `await`
  without violating the refcell rule (take-out-across-await: the genai client/handle is taken out of
  `Interp.resources`, the future is awaited on the owned value, the handle is returned).
- **Bridging fallback (documented, only if the gate fails).** If `genai`'s futures require `Send`:
  run the genai client on a **dedicated single-thread worker** (`std::thread` + its own
  current-thread tokio runtime) and communicate via a channel of owned request/response structs
  (`Value`→neutral-request on the `!Send` side, neutral-response→`Value` back). The worker boundary
  carries only `Send` plain-data structs (strings/bytes/numbers — never `Value`/`Rc`). Streaming
  bridges chunk-by-chunk over an `mpsc`. This preserves the `!Send` core untouched and is the same
  "take-out across the boundary" discipline generalized to a thread. **Cost:** one background thread,
  serialization at the boundary. **Lead with the in-LocalSet path; fall back only on a proven Send
  requirement.**
- **If even the worker proves impractical for one provider** (e.g. a transitively-`Send`-only auth
  client that won't run current-thread): keep `genai` for the providers that work and document the
  affected provider as a Tier-1 "provider unavailable in this build" error — never a silent drop.
  This is not expected; `genai`'s reqwest base makes the in-LocalSet path likely.

**Cargo:** `ai = ["data", "net", "dep:genai"]` (off by default; NOT in `default`). `genai` is added
to `[dependencies]` as `optional = true`. Bedrock/Vertex auth deps come transitively via `genai`'s
`vertex`/`bedrock_sigv4` adapters (no separate `ai-bedrock`/`ai-vertex` sub-features in v1 — owner
decision #2 puts all providers in v1; revisit splitting if the dep weight warrants).

## §2 — `std/ai` API (AScript)

```ascript
import * as ai from "std/ai"
```

### 2.1 Model selection & credentials

A model is `"provider:model"`; credentials resolve from env by default, overridable via an explicit
provider handle.

```ascript
let [out, err] = await ai.generate({
  model: "openai:gpt-4.1",
  prompt: "Write a haiku about Rust ownership.",
})

let [out2, _] = await ai.generate({ model: "anthropic:claude-sonnet-4.5", prompt: "..." })

// OpenAI-compatible endpoint (Ollama / LM Studio / OpenRouter / LiteLLM / xAI / DeepSeek):
let local = ai.provider("openai-compatible", {
  baseUrl: "http://localhost:11434/v1",
  apiKey:  "ollama",
})
let [out3, _] = await ai.generate({ model: local.model("llama3.1"), prompt: "..." })

// Bedrock (SigV4 via AWS credential chain) and Vertex (ADC) — env/instance creds by default:
let [b, _] = await ai.generate({ model: "bedrock:anthropic.claude-sonnet-4.5", prompt: "..." })
let [v, _] = await ai.generate({ model: "vertex:gemini-2.5-flash", prompt: "..." })

// Azure OpenAI / AI Foundry (key auth; deployment-as-model + api-version):
let azure = ai.provider("azure", { baseUrl: "https://my.openai.azure.com", apiKey: env.get("AZURE_OPENAI_KEY"), apiVersion: "2024-10-21" })
let [a, _] = await ai.generate({ model: azure.model("my-gpt4o-deployment"), prompt: "..." })
```

- `ai.provider(kind, config)` → a provider handle (`Value::Native`, state in `Interp.resources`);
  `provider.model(id)` → a model handle. `model:` accepts a `"provider:model"` string (env-resolved)
  OR a model handle (explicit config). Provider kinds map to `genai` adapters:
  `"openai" | "openai-compatible" | "azure" | "anthropic" | "google" | "bedrock" | "vertex"` (plus
  the named OpenAI-compat shortcuts `"ollama" | "openrouter" | "xai" | "deepseek" | "groq" | ...`,
  each a thin preset over `openai-compatible` with a default `baseUrl`). An unknown kind → Tier-2
  misuse panic.
- **Credential precedence** (owner-confirmed env-default): explicit `config.apiKey` › `provider(...)`
  handle › env var (`OPENAI_API_KEY`, `ANTHROPIC_API_KEY`, `GEMINI_API_KEY`, AWS chain for Bedrock,
  ADC/`GOOGLE_APPLICATION_CREDENTIALS` for Vertex, `AZURE_OPENAI_KEY`) › **Tier-1** `[nil,
  {message:"no credential for provider 'openai'"}]` at call time (a missing key is an expected
  operational failure, NOT a panic). genai performs the actual env/credential-chain resolution per
  adapter; `std/ai` maps a genai auth-resolution error to this Tier-1 pair.

### 2.2 Text generation (non-streaming)

```ascript
let [out, err] = await ai.generate({
  model: "anthropic:claude-sonnet-4.5",
  system: "You are a terse assistant.",
  prompt: "Summarize the CAP theorem in one sentence.",
  maxTokens: 256,
  temperature: 0.2,
})
// out == {
//   text, finishReason,                      // "stop"|"length"|"tool_calls"|"content_filter"
//   usage: { inputTokens, outputTokens, totalTokens },
//   toolCalls: [], steps: [], raw: { ... }   // raw = provider-native escape hatch
// }
```

- `prompt` (string) and `messages` (array, §2.6) are mutually exclusive → both set is a Tier-2 panic.
- `system` is hoisted to the provider's system slot by genai per adapter (top-level for Anthropic/
  Gemini, a system message for OpenAI-compat). `maxTokens` is **required by Anthropic** — genai
  surfaces a missing-max-tokens provider error as a Tier-1 `[nil, err]` (we do not silently default).

### 2.3 Streaming (generators + `for await`)

```ascript
let [stream, err] = await ai.stream({ model: "openai:gpt-4.1", prompt: "Explain backpressure." })
if (err != nil) { print(err.message); return }
for await (let chunk in stream) {
  if (chunk.type == "text") { print(chunk.text, { end: "" }) }
}
let final = stream.result()   // { text, finishReason, usage, toolCalls }
```

- `ai.stream(opts)` → Tier-1 `[stream, err]`; `stream` is the established native-handle shape
  (`await stream.next() → [chunk, err]`, consumable by `for await`), backed by `genai`'s
  `exec_chat_stream` mapped chunk-by-chunk (the SSE-reader-style take-out-across-await pattern from
  `net_http.rs`/`SseState`).
- `chunk.type ∈ "text" | "toolCall" | "toolResult" | "finish"` (the AI-SDK `fullStream` projection);
  `"finish"` carries `usage`/`finishReason`. A text-only consumer ignores non-text chunks.
  `stream.textOnly()` returns a convenience generator yielding bare text strings (proposal §2.3
  resolution: one `ai.stream` with typed chunks + a `textOnly()` adapter — NOT a separate
  `streamText`). `stream.result()` (after the loop) returns the terminal aggregate.

### 2.4 Structured / typed output (class or `std/schema` + the JSON-Schema projector)

```ascript
class Recipe { name: string  ingredients: array<string>  steps: array<string> }

let [recipe, err] = await ai.generate({
  model: "openai:gpt-4.1", prompt: "Give me a lasagna recipe.", shape: Recipe,
})
if (err != nil) { print("bad output: " + err.message); return }
print(recipe.name)            // a Recipe instance, fields type-checked

import * as schema from "std/schema"
let s = schema.object({ sentiment: schema.oneOf(["pos","neg","neu"]), score: schema.number() })
let [obj, e] = await ai.generate({ model: "anthropic:...", prompt: "...", shape: s })
```

- **Accept BOTH** a class (`shape: Recipe` → `validate_into`) AND a `std/schema` (owner-confirmed,
  proposal §2.4/§5). The result's `value` is a validated instance / object; a decode failure OR a
  shape mismatch fuses into ONE Tier-1 `[nil, err]`, exactly like `resp.json(Class)`.
- **NEW core mechanism — the class/schema → JSON-Schema projector** (the one genuinely new piece
  SP11 needs; everything else reuses existing code). `validate_into` checks an *already-parsed*
  value, but the provider needs a JSON Schema **up front**. We derive a minimal JSON Schema from a
  class `FieldSchema` (required/optional/nullable/nested-class/`array<Class>`/`map<K,Class>`) and from
  a `std/schema` tagged object, and hand it to genai's structured-output mechanism per provider
  (OpenAI `response_format: json_schema`, Gemini `responseSchema`, Anthropic tool-forcing/JSON). The
  projector lives in `src/stdlib/ai/json_schema.rs` and is unit-tested independently of any network.

### 2.5 Tool calling

```ascript
import * as schema from "std/schema"
let weather = ai.tool({
  description: "Get current weather for a city",
  input: schema.object({ city: schema.string() }),
  execute: async fn(args) {
    let [resp, err] = await getWeather(args.city)
    if (err != nil) { return [nil, err] }       // tool errors are Tier-1, fed back to the model
    return [{ tempC: resp.temp, sky: resp.sky }, nil]
  },
})
let [out, err] = await ai.generate({
  model: "anthropic:claude-sonnet-4.5",
  prompt: "What should I wear in Lisbon today?",
  tools: { weather },
  maxSteps: 5,                                   // cap the tool-use loop
})
// out.text = final answer; out.steps = per-turn description.
```

- A tool = `{ description, input: <schema|class>, execute: fn|async fn }`. The **tool-use loop runs
  inside `std/ai`** (NOT genai's agent layer — we keep control for the Tier-1 semantics): genai
  returns tool calls → we validate args against the tool's schema (the projector, again) → call
  `execute` (awaited if async) → feed results back → loop up to `maxSteps`. **Sequential** in v1
  (non-goal: parallel).
- A tool whose `execute` returns `[nil, err]` surfaces the error back to the model as a tool-result
  (recoverable — owner-confirmed), not an abort. A malformed tool *definition* (no `input`, `execute`
  not callable) → Tier-2 panic.

### 2.6 Message / content model

```ascript
let [out, err] = await ai.generate({
  model: "openai:gpt-4.1",
  messages: [
    { role: "system", content: "You are a vision assistant." },
    { role: "user", content: [
        { type: "text",  text: "What is in this image?" },
        { type: "image", data: imageBytes, mediaType: "image/png" },
    ]},
  ],
})
```

- Roles: `"system" | "user" | "assistant" | "tool"`. Content is a string (text shorthand) or an
  array of typed parts: `"text" | "image" | "file" | "toolCall" | "toolResult"`. genai maps these
  to/from each wire format. Bytes parts use the `Value::Bytes` type; a URL part is a string.

### 2.7 Embeddings

```ascript
let [emb, err]  = await ai.embed({ model: "openai:text-embedding-3-small", value: "sunny day" })
// emb == { embedding: [..], usage: { inputTokens } }
let [embs, err2] = await ai.embedMany({ model: "openai:text-embedding-3-small", values: ["a","b"] })
// embs == { embeddings: [[..],[..]], usage }
```

Backed by genai's `embed` module. Tier-1 `[value, err]` like everything else.

### 2.8 Error model (Tier-1 vs Tier-2)

| Situation | Result |
|---|---|
| Network/TLS/timeout, provider HTTP 4xx/5xx, missing credential, content-filter refusal, structured-output validation failure, missing required param (e.g. Anthropic max_tokens) | **Tier-1** `[nil, err]` (`err.message`, `err.status?`, `err.code?`) |
| Wrong arg *types*, `prompt`+`messages` both set, malformed tool definition, unknown provider kind | **Tier-2** panic |
| Tool `execute` returns `[nil, err]` | Fed back to the model as a tool-result (recoverable), not an abort |

`errorOnStatus`-style default ON for AI calls (a provider error is more useful as a Tier-1 `err` than
a "success" carrying an error body); `raw` is always present as an escape hatch.

## §3 — Telemetry integration (the SP12 soft hook)

- **`std/ai` emits OTel GenAI-convention spans through the SP12 runtime hook** (`Interp::telemetry_*`,
  SP12 §3) — **no Cargo dependency on `telemetry`** (owner decision #3). Each `ai.generate` /
  `ai.stream` / `ai.embed` opens a span named per the GenAI convention
  **`{gen_ai.operation.name} {gen_ai.request.model}`** (e.g. `chat openai:gpt-4.1`,
  `embeddings openai:text-embedding-3-small`) with the literal attribute keys:
  `gen_ai.operation.name`, `gen_ai.provider.name`, `gen_ai.request.model`, `gen_ai.request.temperature`,
  `gen_ai.request.max_tokens`, `gen_ai.request.top_p`, `gen_ai.response.model`,
  `gen_ai.response.finish_reasons`, `gen_ai.response.id`, `gen_ai.usage.input_tokens`,
  `gen_ai.usage.output_tokens`. A tool call → a child span `execute_tool {tool.name}`.
- **Opt-in, runtime-soft.** `std/ai` calls `self.telemetry_active()`; if false (telemetry never
  initialized, or built without the feature), it emits nothing and takes no branch on a Cargo feature.
  When telemetry IS initialized, GenAI spans default ON (owner decision #3: "tracing OPT-IN, activate
  by initializing telemetry"). This makes **Langfuse / any OTel backend light up automatically** via
  the SP12 `otlp` exporter, with no AI-specific exporter.
- **Per-call control + PII safety** (AI-SDK `experimental_telemetry` analog):
  `ai.generate({ ..., telemetry: { enabled: false } })` disables spans for one call;
  `{ recordInputs: true, recordOutputs: true }` opt **in** to recording prompt/response **content**
  (OFF by default — usage/timing/model are always recorded when telemetry is active; content is not,
  to avoid logging PII).

## §4 — Architecture: wrap genai over the `!Send` runtime

```
ai.generate(opts)
  │  parse opts (Value) → genai::chat::ChatRequest + ChatOptions   (Rust)
  │  resolve model handle / "provider:model" → genai ModelIden + creds
  │  shape: → JSON Schema (the projector) → genai structured-output config
  │  tools: → genai tool defs; loop driven by std/ai (Tier-1 semantics), maxSteps
  ▼
genai::Client (custom reqwest injected: pooled rustls client + test send-seam)
  │  exec_chat / exec_chat_stream / embed
  │  run on the current-thread LocalSet (verified Phase A) OR the dedicated worker (fallback §1)
  │  take-out-across-await: client/handle out of Interp.resources, await owned, return
  ▼
genai response → NeutralResponse → Value (out object / stream chunks / instance via validate_into)
  │  emit GenAI span through Interp::telemetry_* (SP12 hook) when telemetry_active()
  ▼
[value, err]  (Tier-1)   |   stream native handle (for await)   |   validated instance
```

- **Stateful handles** (provider, model, stream) live in `Interp.resources`
  (`ResourceState::Ai{Provider, Stream}`) behind `Value::Native` ids — never embedded in `Value`,
  matching the HTTP/SSE precedent.
- **genai client lifetime:** a per-`Interp` genai `Client` (built once, with the injected reqwest
  client) stored in `Interp` (or `resources`); each call takes it out / clones the handle as needed
  for the await, honoring `await_holding_refcell_ref = "deny"`.

## §5 — Testing strategy (no network, no secrets by default)

- **Injectable HTTP-send seam.** genai supports a custom `reqwest::Client`; for fixtures we inject a
  client (or a genai-level seam) that serves **recorded JSON/SSE fixtures** instead of hitting the
  network. Unit tests (default `cargo test --features ai`) replay fixtures under
  `tests/fixtures/ai/<provider>/<case>.{json,sse}` — **no socket, no secret** ever. Cover per
  provider-shape: text generation, streaming (chunk decode + `for await` + `result()`), structured
  output (class + schema, success + validation-failure fusion), the tool-use loop (single + multi
  step, tool error fed back, `maxSteps` cap), embeddings, credential-missing → Tier-1, misuse →
  Tier-2.
- **The JSON-Schema projector** is unit-tested standalone (class → schema, `std/schema` → schema,
  nested/array/map/optional/nullable) with **no network**.
- **The Phase-A `!Send` spike** is a compile+run test asserting genai runs on the LocalSet (or
  switches to the worker bridge) — the gate that de-risks the whole sub-project.
- **Env-gated live suite (CI-excluded by default):** behind `ASCRIPT_AI_LIVE=1` + provider keys, hit
  real providers (small, cheap calls) — one per provider family incl. a Bedrock + a Vertex smoke.
  Never runs without the env vars.
- **Telemetry integration test:** with SP12 initialized (capture mode), assert `ai.generate` emits a
  GenAI span with the correct `gen_ai.*` attributes through the hook (no network — fixture response).
- **Both feature configs + clippy:** `cargo test` (ai OFF by default) passes; `cargo test --features
  ai` exercises the module; `--no-default-features` builds (module cfg'd out). Clippy clean in all
  configs; `await_holding_refcell_ref` denied + clean. **Differential note:** `std/ai` is
  engine-agnostic stdlib (both engines delegate stdlib calls to the same `Interp`), so there is **no
  VM/tree-walker differential surface** — the three-way differential is irrelevant here (stated
  explicitly so the implementer doesn't add spurious differential cases).

## §7 — Open questions (carried to review)

1. **genai `Send`-ness (the load-bearing unknown).** Resolved empirically by the Phase-A gate; the
   worker-bridge fallback (§1) is fully specified if needed. This is the one risk that could change
   the architecture — front-loaded deliberately.
2. **genai's structured-output coverage per provider.** genai's docs emphasize Gemini/Vertex
   structured output; OpenAI `response_format` and Anthropic tool-forcing coverage must be confirmed
   per adapter. Where genai lacks a provider's structured mode, fall back to **prompt-embedded JSON
   Schema + `validate_into` on the text** (always available) — documented, not a silent gap.
3. **Azure AI Foundry vs Azure OpenAI.** v1 covers Azure OpenAI **key auth** (deployment-as-model +
   `api-version`) via genai's openai-compat. Entra-ID/AAD token auth for AI Foundry is a follow-up if
   genai doesn't cover it — flagged at review.
4. **Bedrock/Vertex dep weight.** All providers are in v1 (owner decision #2) via genai's
   `bedrock_sigv4`/`vertex` adapters (transitive AWS/Google auth deps). If the transitive tree proves
   too heavy for the `ai` feature, revisit splitting Bedrock/Vertex behind opt-in sub-features —
   recorded, not done in v1.
